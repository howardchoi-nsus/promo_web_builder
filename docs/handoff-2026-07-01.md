## 오늘 진행 요약

오늘은 `Integrated Design Brief MD`를 실제 이미지 생성의 단일 기준 문서로 만들기 위한 흐름을 end-to-end로 검증하고, Design MD 데이터화 품질 문제까지 진단했다.

핵심 진행 내용은 다음과 같다.

- `Design Prompt MD + Section Input Log MD -> Integrated Design Brief MD -> Image Prompt -> UI Image` 흐름을 n8n Cloud에서 신규 runKey 기준으로 검증했다.
- `promo-031` 기준으로 이미지 asset, Design Prompt MD, Promo Input MD, Integrated Design Brief MD가 각각 Blob asset으로 저장되는 것을 확인했다.
- `promo-design-image`와 `promo-design-view` 조회 링크가 정상 생성되는 것을 확인했다.
- `Integrated Design Brief MD` 생성 시 참조형 표현 금지, Template 4의 7개 섹션 포함, 영어 visible copy 변환, 최종 이미지 생성 지시 포함 여부를 검토했다.
- `LLM Generate Integrated Design Brief` 노드가 120초 timeout에 걸리는 문제를 확인했고, timeout 증가 후 진행되는 것을 확인했다.
- 현재 이미지 품질 저하 원인이 LLM 사양보다 통합문서와 Design Token 매핑 품질에 더 크게 있다는 결론을 냈다.
- Spotify Design MD와 생성된 Design Prompt MD를 비교해 semantic token 매핑 오류를 분석했다.
- 테스트 적용으로 `api/_design-md-data.js`의 Design MD 정규화 로직을 보강했다.

## 문서별 평가

### Design Prompt MD

정상 확인:

- `type: design_prompt`
- `# Design Tokens`
- Colors / Typography / Radius / Spacing / Components / Layouts 포함
- LLM imagePrompt, designBrief JSON, layoutMapping JSON 혼입 없음

발견한 문제:

- Spotify 기준으로 `background: #ffffff`, `text: #000000`처럼 잘못된 semantic role 매핑이 있었다.
- `spacing.base`가 원래 `8px`이어야 하는데 `1.4`로 잡혔다.
- `bodyFont`, `displayFont`, `radius.pill` 같은 핵심 상위 토큰이 `unknown`으로 남았다.
- 원본 문서의 깨진 범위 문자(`??`)가 일부 그대로 전파됐다.

### Promo Input MD

정상 확인:

- `type: section_input_log`
- Promo / Simple Brief / Section Inputs / Template / Design Style 포함
- 후속 생성물 혼입 없음

주의점:

- 입력 로그 문서에는 한글 원본 visible copy가 남아 있다.
- 이 자체는 입력 보존 목적상 허용 가능하나, Integrated Brief에서는 영어로 변환되어야 한다.

### Integrated Design Brief MD

정상 확인:

- `# Integrated Design Brief MD`
- 필수 섹션 포함
- Template 4의 7개 섹션 포함
- `Design Token Application` 포함
- `Token-to-Section Application` 포함
- `Final Image Prompt Inputs` 포함
- `Negative Prompt` 포함
- `Visual QA Checklist` 포함
- 한글 visible copy 없음
- 금지 참조 표현 없음

수정 필요:

```text
현재: type: design_prompt
필요: type: integrated_design_brief
```

품질 개선 필요:

- 내용이 아직 `Standard height`, `Clear and concise`, `Dark background with white text`처럼 추상적인 표현에 머무는 경우가 있다.
- section별 UI 설계 밀도가 부족하다.
- Design Token 값이 실제 섹션별 레이아웃과 컴포넌트 지시로 충분히 변환되지 않는다.

## 오늘 적용한 코드 변경

수정 파일:

```text
api/_design-md-data.js
```

적용 목적:

Spotify Design MD를 테스트 대상으로 semantic token 매핑을 개선했다.

개선 전 주요 문제:

```text
background: #ffffff
text: #000000
surface: unknown
mutedText: unknown
accent: #1f1f1f
spacing.base: 1.4
radius.pill: unknown
bodyFont: unknown
depthModel: glass
```

개선 후 로컬 재추출 결과:

```text
primary: #1ed760
accent: #1ed760
background: #121212
surface: #181818
text: #ffffff
mutedText: #b3b3b3
border: #4d4d4d
displayFont: SpotifyMixUITitle
bodyFont: SpotifyMixUI
titleWeight: 700
spacing.base: 8px
radius.pill: 9999px
radius.small: 4px
radius.medium: 6px
radius.large: 100px
cardShadow: rgba(0,0,0,0.5) 0px 8px 24px
depthModel: shadowed
```

검증:

```text
node --check api/_design-md-data.js: pass
scripts/test-design-prompt-md.js: pass
Spotify Design MD local extraction: pass
```

주의:

- 코드 로직은 수정되었지만 DB에 저장된 기존 Spotify 토큰은 아직 자동 변경되지 않았다.
- DB 반영을 위해서는 앱 UI에서 Spotify 문서를 재추출하거나, `DATABASE_URL` 설정 후 import script를 실행해야 한다.
