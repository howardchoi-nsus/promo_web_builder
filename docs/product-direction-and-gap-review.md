# Promo AI Builder 제품 방향 및 구현 검토

## 서비스 개발 배경

현재 운영 중인 프로모션 페이지 빌더는 AI 기반이 아닙니다.

기존 구조는 몇 가지 고정 템플릿을 제공하고, 각 리전 담당자가 리전 상황에 맞게 텍스트나 콘텐츠를 수정하면서 프로모션 페이지를 업데이트하고 운영하는 방식입니다.

하지만 운영 과정에서 지속적으로 새로운 디자인 추가, 즉 템플릿 추가 요구가 발생하고 있습니다. 매번 디자인 템플릿을 수동으로 추가하는 방식은 확장성이 낮고, 디자인 다양성 요구를 빠르게 따라가기 어렵습니다.

따라서 AI를 도입해 디자인 생성 과정을 자동화하고, 최종적으로 퍼블리싱까지 연결되는 프로모션 빌더를 만드는 것이 목표입니다.

## 만들고자 하는 방향

목표 서비스는 단순한 페이지 생성기가 아니라, 기존 운영형 프로모션 빌더에 AI 디자인/퍼블리싱 레이어를 얹는 구조입니다.

핵심 방향은 아래와 같습니다.

```text
기존 프로모션 운영 요구
  -> 리전 담당자 입력
  -> 디자인 MD 기반 디자인 생성
  -> 생성 디자인 확인 및 수정
  -> 코드 생성
  -> 자동 퍼블리싱
```

구체적으로는:

- 디자인 MD를 이용해 다양한 디자인 톤앤매너, 레이아웃, 디자인 콘셉트를 지원받음.
- B섹션에서 입력되는 프로모션 내용을 참고해 AI가 디자인을 생성함.
- 생성된 디자인을 운영자가 확인하고 수정함.
- 승인된 디자인을 기준으로 프론트 코드로 변환함.
- 최종적으로 퍼블리싱까지 자동화함.

## 현재 구현된 범위

현재 구현은 아래 흐름까지 진행되어 있습니다.

```text
디자인 MD 선택
  -> 프로모션 정보 입력
  -> Temp.4 섹션 초안 생성
  -> UI 디자인 이미지 생성
  -> 결과 확인
```

### A섹션

- 디자인 MD 목록을 표시함.
- MD를 단순 목록이 아니라 `Design Style Concept` 기준으로 그룹핑함.
- 스타일 그룹 선택 후 하위 MD 리스트가 노출됨.
- 태그, confidence, 미분류 상태를 확인할 수 있음.

### B섹션

- 프로모션 입력 구조를 Temp.4 기준으로 정리함.
- 간편 프로모션 브리프 입력 구조가 있음.
- AI 섹션 초안 생성 흐름이 있음.
- 아래 Temp.4 섹션 입력 구조를 반영함.
  - Hero Banner
  - Step Bar
  - Content CTA
  - Image Text Row
  - Terms / Footer
- 색상/폰트 스타일 선택 및 재정의 흐름이 있음.

### C섹션

- 생성 결과 목록을 표시함.
- 최근 생성 결과를 확인할 수 있음.
- n8n에서 반환하는 `designUrl`, `imageUrl`, `layoutMapping`, `mdComplianceMap`을 받을 수 있는 구조가 있음.

### n8n workflow

- `promo-ui-design-image-generator.workflow.json` 기준으로 UI 디자인 이미지 생성 흐름을 구성함.
- 선택된 MD, Design Concept, Layout Philosophy, Color/Font, Promo 입력값, Temp.4 구조가 prompt에 반영되도록 개선함.
- MD 디자인 철학 반영 여부를 확인할 수 있도록 `mdComplianceMap` 방향을 강화함.

## 현재 구현에 대한 평가

현재 구현은 목표 서비스의 첫 번째 단계로는 적절합니다.

특히 아래 핵심 가설은 이미 코드에 반영되어 있습니다.

```text
디자인 MD를 선택하고,
B섹션 입력값을 기반으로,
AI가 UI 디자인 이미지를 생성한다.
```

즉, 현재는 `AI 디자인 생성 실험 단계`까지 구현된 상태입니다.

다만 최종 목표가 `AI 기반 프로모션 디자인/퍼블리싱 빌더`라면, 아직 아래 단계는 부족합니다.

```text
생성 디자인 수정
  -> 승인
  -> 코드 변환
  -> 정책 검증
  -> 저장
  -> 퍼블리싱
```

## 보완이 필요한 부분

### 1. 기존 운영 빌더와의 연결 구조

현재 구현은 독립 POC에 가깝습니다.

실제 서비스 목표는 기존 프로모션 빌더를 확장하거나 대체하는 것이므로, 기존 운영 빌더의 개념이 더 들어와야 합니다.

보완 필요 항목:

- 기존 템플릿 목록
- 기존 템플릿 필드 구조
- 리전별 담당자 입력 항목
- 기존 프로모션 페이지 데이터 구조
- 퍼블리싱 대상 경로
- 승인/검수 상태

현재는 `Template 4` 하나를 고정으로 다루고 있습니다. 운영화하려면 템플릿과 필드를 DB 구조로 분리해야 합니다.

### 2. 디자인 MD와 템플릿의 관계 강화

현재 MD는 디자인 톤앤매너 참고자료 역할에 가깝습니다.

하지만 목표 서비스에서는 MD가 단순 참고자료가 아니라, 디자인 생성에 직접 영향을 주는 디자인 시스템/레이아웃 지시서로 작동해야 합니다.

현재 구조:

```text
MD 선택
  -> prompt에 포함
```

보완 방향:

```text
MD
  -> layout rules
  -> component rules
  -> forbidden patterns
  -> section composition
  -> responsive rules
  -> publishing constraints
```

특히 `layoutPhilosophy`, `componentStyle`, `depthAndEffects`, `dos/donts`는 단순 설명이 아니라 생성 결과 검증 조건으로 사용해야 합니다.

### 3. 생성 결과 수정 흐름

현재 C섹션은 생성 결과를 확인하는 단계에 가깝습니다.

하지만 실제 목표는 생성된 디자인을 확인하고 수정한 뒤 퍼블리싱하는 것입니다.

필요한 흐름:

```text
생성 결과 확인
  -> 수정 요청 입력
  -> 재생성 또는 부분 수정
  -> 버전 비교
  -> 승인
  -> 코드 생성
  -> 퍼블리싱
```

C섹션에 추가하면 좋은 기능:

- 생성 이미지 미리보기
- MD compliance 결과 표시
- 수정 요청 입력
- 재생성 버튼
- 승인 버튼
- 코드 생성 버튼
- 퍼블리싱 버튼
- 생성 이력 / 버전 관리

### 4. 이미지 생성과 프론트 코드 생성 분리

현재는 UI 디자인 이미지 생성 단계입니다.

자동 퍼블리싱까지 가려면 이미지 생성 이후에 별도 코드 생성 단계가 필요합니다.

추천 흐름:

```text
1. 디자인 이미지 생성
2. 운영자 확인 / 수정
3. 승인된 디자인 기준으로 HTML/CSS/Vue 생성
4. 미리보기 QA
5. 정책 / 약관 검증
6. 퍼블리싱
```

n8n workflow도 아래처럼 분리하는 것이 좋습니다.

- `promo-ui-design-generate`
- `promo-ui-design-revise`
- `promo-code-generate`
- `promo-publish`
- `promo-run-status`

### 5. 운영 정책 검증

프로모션 페이지는 디자인뿐 아니라 운영 안전성이 중요합니다.

특히 리전 담당자가 직접 사용하는 빌더라면, AI 생성 전에 Rule Base 검증이 필요합니다.

검증이 필요한 항목:

- 리전별 필수 약관
- Responsible Gaming 문구
- Bonus Code
- Affiliate
- Q-TAG
- 만 18세/21세 고지
- 국가별 금지 표현
- CTA 링크 정책
- 기간/조건 필수 입력
- 법무/컴플라이언스 체크

현재는 Terms 입력란이 있지만, Rule Base 검증은 아직 약합니다.

AI가 디자인을 만들더라도 운영 정책은 Rule Base가 통제해야 합니다.

### 6. DB 저장 구조

현재 생성 결과는 localStorage와 n8n static data 중심입니다.

운영화하려면 생성 이력과 결과를 DB에 저장해야 합니다.

검토 후보 테이블:

- `promo_templates`
- `promo_template_fields`
- `promo_design_runs`
- `promo_design_assets`
- `promo_design_versions`
- `promo_code_outputs`
- `promo_publish_jobs`
- `promo_region_rules`
- `promo_compliance_checks`

특히 `promo_design_runs`는 우선순위가 높습니다.

저장해야 할 정보:

- 누가 생성했는지
- 어떤 리전/마켓인지
- 어떤 MD를 사용했는지
- 어떤 입력값을 사용했는지
- 어떤 prompt를 사용했는지
- 어떤 이미지가 생성됐는지
- 어떤 compliance 결과가 나왔는지
- 승인 여부
- 코드 생성 여부
- 퍼블리싱 여부

### 7. Advanced Mode 고도화

현재 Simple Mode 방향은 적절합니다.

비전문가용으로 `간편 브리프 -> AI 섹션 초안` 흐름은 유지하는 것이 좋습니다.

다만 Advanced Mode는 아직 실질 UX가 약합니다.

추천 구조:

```text
Simple Mode:
  - 제목
  - 주요 혜택
  - 유도 행동
  - CTA
  - 대상 고객
  - 약관

Advanced Mode:
  - Header 직접 수정
  - Hero Banner 직접 수정
  - Step Bar 직접 수정
  - Content CTA 직접 수정
  - Image Text Row 직접 수정
  - Terms/Footer 직접 수정
  - 섹션별 visualMode 설정
  - 섹션별 생성/잠금
```

특히 `섹션 잠금` 기능이 중요합니다.

운영자는 특정 문구나 약관은 고정하고, 디자인만 다시 생성하고 싶을 수 있습니다.

## 추천 개발 우선순위

### 1순위: 제품 목적 문서화

README 또는 docs에 아래 방향을 명확히 추가해야 합니다.

```text
기존 템플릿 기반 프로모션 빌더
  + AI 디자인 자동화
  + 운영 검수
  + 코드 생성
  + 자동 퍼블리싱
```

### 2순위: C섹션 확장

C섹션은 단순 결과 확인 영역이 아니라 `검수 및 다음 액션` 영역이 되어야 합니다.

추가할 기능:

- 이미지 미리보기
- MD compliance 표시
- 수정 요청
- 재생성
- 승인
- 코드 생성
- 퍼블리싱

### 3순위: 생성 workflow 단계 분리

현재 `UI 디자인 이미지 생성` 단계와 앞으로 필요한 `코드 생성`, `퍼블리싱` 단계를 분리해야 합니다.

```text
Design Generate
  -> Design Revise / Approve
  -> Code Generate
  -> Publish
```

### 4순위: DB 모델 초안 추가

최소 아래 테이블부터 검토하는 것이 좋습니다.

- `promo_design_runs`
- `promo_design_assets`
- `promo_publish_jobs`

### 5순위: 리전/정책 Rule Base 추가

AI 생성보다 먼저 운영 안전장치가 필요합니다.

정책 검증은 AI prompt가 아니라 Rule Base로 처리하는 것이 안전합니다.

## 결론

현재 구현은 목표 서비스의 첫 번째 단계로는 잘 맞습니다.

지금까지는 `디자인 MD + 프로모션 입력값 -> UI 디자인 이미지 생성`이라는 핵심 가설을 검증하는 단계입니다.

하지만 최종적으로 만들고 싶은 서비스는 아래에 가깝습니다.

```text
운영자가 쓰는 AI 기반 프로모션 디자인/퍼블리싱 빌더
```

따라서 다음 개발은 C섹션을 `결과 확인`이 아니라 `검수 및 다음 액션` 영역으로 확장하는 것이 가장 좋습니다.

이후에는 `수정`, `승인`, `코드 생성`, `정책 검증`, `퍼블리싱` 흐름을 순서대로 붙이는 것이 자연스럽습니다.
