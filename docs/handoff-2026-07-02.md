# Handoff - 2026-07-02

## 오늘 진행 요약

오늘은 C섹션 생성 이미지 목록을 서버 기준으로 보는 구조에서 발생한 이미지 표시 실패, n8n Gemini 이미지 생성 전환, 이미지 저장 안정화, 출력 해상도 이슈를 중심으로 디버깅했다.

핵심 결론은 다음과 같다.

- `Generate an image` 노드에서 이미지 preview가 가능하면 이미지 생성 자체는 성공이다.
- 과거 `file_size: 9` 이미지는 생성 실패가 아니라 n8n Store 단계에서 binary 참조값을 실제 이미지처럼 읽어 서버에 저장한 문제였다.
- 서버는 기존에 invalid image bytes를 저장 전에 막지 못했기 때문에 깨진 이미지가 C섹션에 누적될 수 있었다.
- `gemini-3-pro-image`는 5~7분대 timeout이 반복되어 운영 기본 모델로 부적합했다.
- `gemini-3.1-flash` 계열은 약 1분 27초에 성공해서 운영 기본 모델 후보로 적합했다.
- Gemini 기본 n8n 노드는 해상도/aspect ratio 옵션을 충분히 노출하지 않아 `848x1264` 같은 1K 2:3 결과가 생성됐다.
- 해상도 문제는 리사이즈보다 Gemini Interactions API HTTP Request 직접 호출로 `response_format.aspect_ratio`, `response_format.image_size`를 지정하는 방향이 더 적합하다.

## 오늘 확인한 주요 이슈

### 1. C섹션 이미지가 보이지 않음

증상:

```text
Generated image could not be loaded. Check the image asset or Blob access for this design.
```

라이브 API 확인 결과:

```text
promo-2607021422-wE84v
mime_type: image/jpeg
file_size: 9

promo-2607021219-h8O9r
mime_type: image/jpeg
file_size: 9

promo-2607021049-SdhM8
mime_type: image/png
file_size: 2068465
```

판정:

```text
프론트/C섹션 렌더링 문제가 아니라 서버 Blob/DB에 이미 깨진 이미지가 저장된 상태.
```

원인:

```text
n8n Cloud binaryDataMode: filesystem
-> currentItem.binary.xxx.data가 실제 이미지 base64가 아닌 참조값일 수 있음
-> Store UI Design Result에서 이를 base64처럼 처리
-> 9 bytes 이미지로 저장
```

### 2. 서버 저장 방어 부족

기존 서버는 `imageDataUrl`, `b64Json`, `imageUrl`에서 받은 bytes가 실제 PNG/JPEG/WebP인지 강제 검증하지 않았다.

반영된 개발 방향:

- `/api/promo-design-assets` 저장 전에 image signature 검증
- invalid bytes는 `400 Invalid image data`로 거부
- `/api/promo-design-image` 조회 시 깨진 Blob이면 `422 Invalid stored image` 반환
- C섹션에서 `file_size < 1024` 또는 비이미지 MIME이면 `이미지 손상 의심` 표시

검증:

```text
invalid image POST -> 400 Invalid image data
node --check api/promo-design-assets.js: pass
node --check api/promo-design-image.js: pass
tracked secret pattern 없음
```

### 3. Gemini 모델 timeout

발생한 timeout:

```text
gemini-3-pro-image
7분 21초 실패
5분 실패
6분 15초 실패
```

동일 계열 테스트:

```text
gemini-3.1-flash
1분 27초 성공
```

판정:

```text
imagePromptLength 3036 기준으로 프롬프트 길이 과부하보다는 모델 처리시간/안정성 문제가 더 큼.
```

권장 운영 정책:

```text
기본 모델: gemini-3.1-flash-image
고품질 옵션: gemini-3-pro-image
fallback: pro 실패 시 flash
```

### 4. Integrated Brief JSON parse 실패

발생 에러:

```text
Failed to parse integrated design brief JSON
Unterminated string in JSON at position 33896
```

확인값:

```text
contentLength: 33896
```

재시도 결과:

```text
수정 없이 다시 요청하니 약 12000자 정도로 정상 생성
```

판정:

```text
항상 재현되는 구조적 오류라기보다 LLM 출력 변동성 문제.
간헐적으로 Integrated Brief가 과도하게 길어지고 JSON이 끝에서 잘림.
```

권장:

- Integrated Brief 응답 길이 guard 추가
- 실패 시 동일 입력 1회 자동 재시도
- 15000자 이상은 warning, 30000자 이상은 실패 가능 범위로 판단

### 5. 출력 해상도 848x1264 이슈

사용자는 최소 가로 1024px를 기대했으나 실제 생성 이미지가 약 `848x1264`로 생성됐다.

원인:

```text
프론트/Normalize Payload에는 output.imageSize = 1024x1536이 존재
하지만 n8n Gemini 기본 노드 Generate an image는 options: {} 상태
즉 1024x1536이 실제 API 파라미터로 전달되지 않음
```

중요:

```text
프롬프트 안의 "1024x1536"은 모델 지시일 뿐, 픽셀 크기 강제가 아님.
```

공식 Gemini 이미지 문서 기준:

```text
gemini-3.1-flash-image
2:3 + 1K = 848x1264
2:3 + 2K = 1696x2528
```

따라서 `848x1264`는 Gemini의 `2:3 + 1K` 기본 결과로 보는 것이 타당하다.

권장 해결:

```text
n8n Gemini 기본 노드 대신 HTTP Request로 Gemini Interactions API 직접 호출
response_format.aspect_ratio = "2:3"
response_format.image_size = "2K"
```

리사이즈는 권장 1순위가 아니다. 단순 업스케일은 UI 텍스트/라인이 흐려질 수 있으므로 원본 생성 단계에서 2K로 생성하는 것이 더 낫다.

## n8n HTTP Request 전환 작업

대상:

```text
Generate an image / Generate an image2
```

기존:

```text
@n8n/n8n-nodes-langchain.googleGemini
resource: image
modelId: models/gemini-3.1-flash-image
prompt: {{$json.imagePrompt}}
options: {}
```

문제:

```text
aspect_ratio, image_size를 실제 API 파라미터로 보낼 수 없음.
```

수정 방향:

```text
n8n-nodes-base.httpRequest
POST https://generativelanguage.googleapis.com/v1beta/interactions
```

HTTP Request Body:

```js
={{
  JSON.stringify({
    model: String($json.imageModel || 'gemini-3.1-flash-image').replace(/^models\//, ''),
    input: [
      {
        type: 'text',
        text: $json.imagePrompt
      }
    ],
    response_format: {
      type: 'image',
      mime_type: 'image/jpeg',
      aspect_ratio: $json.output?.aspectRatio || '2:3',
      image_size: $json.output?.imageSizeTier || '2K'
    }
  })
}}
```

주의:

- `mime_type: image/png`은 현재 API에서 거부됨.
- 에러 메시지 기준 지원값은 `image/jpeg`.
- 따라서 `mime_type`은 일단 `image/jpeg`로 고정해야 한다.

HTTP Request Header:

```json
{
  "x-goog-api-key": "새로_발급한_Gemini_API_Key",
  "Content-Type": "application/json"
}
```

또는 `Using Fields` 방식:

```text
Name: x-goog-api-key
Value: 새로_발급한_Gemini_API_Key

Name: Content-Type
Value: application/json
```

주의:

- `name: projects/877687115837`처럼 넣으면 안 된다.
- `name`은 header 이름이어야 한다.
- n8n Cloud에서는 `$env.GEMINI_API_KEY` 접근이 차단되어 있음.
- `{{$env.GEMINI_API_KEY || ...}}` 표현식은 fallback 이전에 에러가 난다.
- API key가 채팅/로그에 노출됐으므로 기존 키는 폐기하고 새 키를 발급해야 한다.

## 현재 남은 문제

### 1. HTTP Request Gemini API 최종 성공 확인 필요

마지막 진행 상태:

```text
403: API key/header 문제 해결
400: image/png unsupported 확인
다음 조치: mime_type을 image/jpeg로 고정 후 재실행 필요
```

다음 실행에서 확인할 값:

```text
Generate an image2 성공 여부
응답에 output_image.data 또는 steps[].content[].data 포함 여부
Store UI Design Result imageBytes
Persist 후 /api/promo-design-assets file_size
실제 이미지 dimensions
```

기대값:

```text
aspect_ratio: 2:3
image_size: 2K
예상 출력: 1696x2528 계열
최소 가로 1024px 충족
```

### 2. Store UI Design Result 응답 파싱 확인 필요

Interactions API 응답은 기존 Gemini n8n 노드와 구조가 다르다.

보강해야 하는/보강한 파싱 경로:

```js
response.output_image?.data
response.outputImage?.data
response.steps[].content[].data
response.candidates[].content.parts[].inlineData.data
$input.item.binary via getBinaryDataBuffer
```

다음 테스트에서 Store 노드가 `imageBytes > 10000`을 내는지 확인해야 한다.

### 3. 기존 깨진 산출물 처리

기존에 저장된 `file_size: 9` 이미지는 복구할 수 없다.

처리 후보:

```text
옵션 A: DB status invalid 처리
옵션 B: C섹션에서 경고 표시만 유지
옵션 C: 삭제
```

추천:

```text
옵션 A 또는 B
```

삭제보다 추적 가능성이 좋다.

### 4. Integrated Brief 길이 안정화

간헐적으로 33000자 이상 응답이 나와 JSON parse가 실패했다.

권장:

```text
contentLength > 15000 warning
contentLength > 20000 retry
contentLength > 30000 fail/retry
```

프롬프트에도 compact 제한을 추가하는 것이 좋다.
