# Handoff - 2026-06-30

## 오늘의 핵심 목표

선택된 Design MD와 B섹션 프로모션 요구사항 로그를 통합해 `Integrated Design Brief MD`를 만들고, 이 통합문서만 기준으로 UI 디자인 이미지 생성 요청이 이어지도록 구조를 정리했다.

## 주요 논의 정리

- `Integrated Design Brief MD`는 요약문이 아니라, downstream LLM이 이 문서만 보고 UI 디자인 생성 프롬프트를 만들 수 있는 독립 산출물이어야 한다.
- 기존 산출물은 `promo-integrated-design-brief-generation.md`의 의도와 다르게 느슨하게 생성될 수 있었다.
- 원인은 프롬프트 문서가 `type`, YAML frontmatter, sourceDocuments, height 정책을 강하게 고정하지 않았고, n8n 검증도 heading/길이 중심으로만 되어 있었기 때문이다.
- height 정책은 `1024x1536` 고정도 아니고 `3000px` 제한도 아니다.
- 최종 기준은 `1440px` desktop page width 기준, height는 콘텐츠에 따라 자유롭게 늘거나 줄며, Header부터 Footer까지 잘리지 않는 full-page Web UI design mockup이다.

## 다음 논의 필요 - Integrated Design Brief MD 정의 불일치

### 사용자 요구사항 재확인

사용자가 원하는 `Integrated Design Brief MD`는 단순 저장/조회용 세 번째 문서가 아니다.

정의:

```text
Design Prompt MD + Section Input Log MD = Integrated Design Brief MD
```

즉, 최종 `Integrated Design Brief MD`는 앞의 두 문서를 다시 함께 봐야 이해되는 보조 문서가 아니라, downstream LLM이 이 문서 하나만 읽고 웹 디자인 이미지를 만들 수 있는 독립 실행 문서여야 한다.

반드시 포함해야 할 것:

- 선택된 Design MD의 디자인 철학, 레이아웃, 컴포넌트 규칙
- B섹션 입력값 전체
- 섹션별 실제 visible copy
- Template 4 섹션 순서와 매핑
- 충돌 해결 결과
- 최종 이미지 생성 지시
- 이 문서 하나만 읽고도 LLM이 웹 디자인 이미지를 만들 수 있는 수준의 상세 지시

### 현재 구현의 한계

현재 구현은 `Integrated Design Brief MD`를 저장하고 조회하는 기능은 붙었지만, 문서 생성 방식이 사용자 요구와 다르다.

현재 흐름:

```text
designBrief JSON + layoutMapping JSON -> Integrated Design Brief MD
```

문제:

- 실제 저장되는 `Design Prompt MD`와 `Section Input Log MD` 전문을 합쳐서 integrated brief를 만들지 않는다.
- integrated brief 생성 프롬프트는 self-contained를 요구하지만, 입력 자체가 두 MD 문서 전문이 아니라 중간 JSON이다.
- 결과적으로 생성된 integrated brief가 앞의 두 문서를 완전히 대체하지 못하고, 참조형/요약형 문서처럼 보일 수 있다.
- `Design Prompt MD` 안에 `Integrated Design Brief` 섹션이 들어가는 구조라 방향이 반대로 느껴질 수 있다.

### 원하는 목표 흐름

내일 논의할 기준 흐름:

```text
1. Design Prompt MD 초안 생성
2. Section Input Log MD 생성
3. 두 MD 전문을 Integrated Design Brief Generation 노드에 입력
4. LLM이 두 문서를 통합해 독립형 Integrated Design Brief MD 생성
5. 이미지 프롬프트 생성 LLM은 오직 Integrated Design Brief MD만 읽음
6. generated image + 세 MD 문서를 각각 저장
```

중요한 원칙:

- 이미지 생성 단계는 `Design Prompt MD`, `Section Input Log MD`, `designBrief JSON`, `layoutMapping JSON`을 다시 읽으면 안 된다.
- Integrated brief 안에 모든 최종 판단과 디자인/콘텐츠 결정이 재서술되어야 한다.
- 문서 안에 "see Design Prompt MD", "refer to Section Input Log" 같은 참조형 표현이 있으면 실패로 봐야 한다.

### 필요한 수정 방향

다음 작업 후보:

1. `api/promo-design-assets.js`의 markdown 생성 책임 재검토
   - 현재는 저장 시점에 `buildDesignPromptMarkdown`, `buildPromoInputMarkdown`을 만든다.
   - integrated brief 생성이 n8n 앞단에서 두 MD 전문을 입력받으려면, 두 MD를 저장 API 안에서만 만들 것이 아니라 n8n 단계에서도 동일한 markdown 텍스트를 생성하거나 API로 사전 생성할 방법이 필요하다.

2. `n8n/promo-ui-design-image-generator.workflow.json` 재설계
   - `Build Integrated Design Brief Request` 노드 입력을 `designBrief/layoutMapping JSON` 중심에서 `Design Prompt MD full text + Section Input Log MD full text` 중심으로 변경.
   - `Parse Validate Integrated Design Brief`에서 참조형 문구 금지 검증 추가.
   - integrated brief 길이뿐 아니라 필수 copy/section/design decision 포함 여부를 검증.

3. `prompts/promo-integrated-design-brief-generation.md` 강화
   - "Do not summarize" 명시.
   - "Do not say refer to Design Prompt MD or Section Input Log" 명시.
   - 두 입력 문서의 모든 최종 디자인/콘텐츠 결정을 integrated brief에 복사, 해석, 충돌 해결, 재서술하도록 지시.

4. C섹션 표시 정책 정리
   - `Design Prompt MD`
   - `Section Input Log MD`
   - `Integrated Design Brief MD`
   세 문서의 역할이 사용자에게 명확히 보이도록 라벨을 정리할 필요가 있다.

### 내일 시작 질문

- `Design Prompt MD`와 `Section Input Log MD`를 n8n 내부에서 먼저 markdown 문자열로 생성할지, 아니면 API에서 markdown preview/generate endpoint를 추가할지 결정해야 한다.
- `Integrated Design Brief MD`의 필수 구조를 지금 11개 섹션 구조로 유지할지, "Design Prompt + Section Input 통합문서"에 맞춰 재구성할지 결정해야 한다.
- 기존 `designBrief JSON`, `layoutMapping JSON`은 integrated brief 생성의 보조 입력으로 유지할지, 아니면 두 MD 문서 안에 포함된 정보로만 제한할지 결정해야 한다.

## 남은 리스크

- 이미지 모델의 실제 bitmap size는 API가 허용하는 값으로 제한될 수 있다.
- 따라서 “height 자유”는 실제 픽셀 캔버스가 무제한이라는 뜻이 아니라, 프롬프트에서 full-page UI를 crop 없이 축소 표현하도록 유도하는 정책이다.

- 브라우저 localStorage에 기존 실패 상태가 남아 있으면 C섹션 목록이 일시적으로 실패처럼 보일 수 있다. 이 경우 강력 새로고침 후 `열기` 또는 `MD 보기`를 다시 누르면 저장 결과 복구 로직이 동작한다.
- 기존 runKey는 integrated brief 별도 asset이 없을 수 있으며, 이 경우 `design_prompt_markdown` fallback으로 조회된다.
